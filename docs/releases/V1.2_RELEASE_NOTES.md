## Known limitations and accepted risks (carried into V1.2, not hidden)

* **No real alert-delivery channel exists** (Slack/email/PagerDuty) — SLI
  thresholds are defined and queryable via `GET /api/v1/ops/dashboard`,
  but nothing pages a human automatically yet. Owner decision pending.
* **No real Cloudflare Rate Limiting Rules configured** — the
  architecturally-correct choice (platform feature, not app code, since
  the deploy target is Cloudflare Workers) was made in `V12-01`, but the
  actual dashboard configuration is an owner action not yet taken.
* **Email verification / forgot-password do not exist** in the shared
  legacy `4.php` member system this project's BFF fronts — flagged as a
  separate, independent follow-up task, out of this project's own scope
  to fix directly (that system serves ~12 unrelated products).
* **The shared `4.php` system has known security issues** (plaintext DB
  password, a SQL-leaking debug endpoint, account enumeration) — also
  flagged as a separate follow-up, not fixed here (out of scope: never
  modify the shared cross-product system).
* **7 residual `npm audit` findings**, all in build/dev-toolchain
  transitive dependencies (`wrangler`/`miniflare`/`@opennextjs/*`/`next`
  via `postcss`/`sharp`) — none reachable in the deployed runtime bundle;
  accepted, see `V12-05`'s risk register for the full rationale.
* **No API request-level latency/error log table exists** — Frontend gets
  this for free from Cloudflare's own Worker analytics; Backend relies on
  the shared host's PHP error log (reviewed via hosting control panel).
* **PHP-side (`audit_logs.request_id`) and Next.js-side (`correlationId`)
  tracing are two separate mechanisms**, not unified across the trust
  boundary yet.
* **AEO/GEO Signal detection has no scan-history persistence layer** —
  GA/SEO domain adapters are proven; AEO/GEO are explicitly deferred.
* **No real pilot has been conducted** (`V12-07` — blocked on having a
  real deploy and real consented participants; complete protocol is
  ready).
* **Old OAuth secret and other V08-02-era credentials remain a possible
  exposure** — owner has explicitly accepted this risk rather than force
  an external rotation.
* **PHP 7.0 target / URL-only linting** — an accepted risk from earlier in
  the roadmap, not revisited in V1.2.
